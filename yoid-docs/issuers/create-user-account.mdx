## Overview

The first step in the issuance workflow is creating a youth account. This provisions a Keycloak identity in the **Consumers** realm and initialises a wallet tenant — giving the youth a place to receive and store credentials.

<Note>
  This is more than "creating a user." The API provisions a full wallet tenant backed by the Credo SSI framework. If the call fails, the most common cause is a duplicate email — the youth may already have an account from another partner's onboarding.
</Note>

## Endpoint

```
POST https://test.didxtech.com/consumer-onboarding/api/users
```

## Request

```bash
curl -X POST "https://test.didxtech.com/consumer-onboarding/api/users" \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "email": "[email protected]",
    "firstName": "Thandi",
    "lastName": "Molefe"
  }'
```

### Request Body

| Field | Type | Description |
|-------|------|-------------|
| `email` | string | Youth's email address — must be unique across the platform |
| `firstName` | string | Youth's first name |
| `lastName` | string | Youth's last name |

## Response

**HTTP 201 Created**

```json
{
  "email": "[email protected]",
  "tempPassword": "!w&3WgYeEGMv"
}
```

### Response Fields

| Field | Description |
|-------|-------------|
| `email` | The youth's email address |
| `tempPassword` | A temporary password generated for the youth |

## What Happens on the Youth's Side

1. A Keycloak account is created in the **Consumers** realm
2. A wallet tenant is initialised — this is where credentials will be stored
3. A temporary password is generated and communicated to the youth via email
4. The youth must update this password on first login
5. Once they sign in, any previously issued credentials appear as pending items waiting for acceptance

<Warning>
  **Async pipeline**: You don't need to wait for the youth to activate their account before issuing credentials. Credentials issued before activation are held in custody and delivered automatically when the youth signs in.
</Warning>

## Handling Existing Accounts

If the youth already has an account (created by another partner), the API will return an error. In the YOMA ecosystem, youth frequently interact with multiple partners — Thandi might be onboarded by both Umuzi and JA Worldwide. The account only needs to be created once.

Check your error handling for `400 Bad Request` responses indicating a duplicate email.

<Tip>
  Proceed to [Step 2: Create Credential Template](/api/issuers/create-credential-template) to define the structure of the credential you want to issue.
</Tip>
